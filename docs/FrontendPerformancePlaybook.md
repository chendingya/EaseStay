# 前端性能优化复用手册

## 1. 适用范围
- 适用于 Admin、运营后台、商户后台、以及所有 React 页面。
- 目标是沉淀跨页面可复用规则，不绑定某一个页面。
- 优化必须保证可回归：`build` 与 `test` 必须一起通过。

## 2. 通用优化原则

### 2.1 加载原则
- 首屏只加载首屏需要的功能代码。
- 低频功能延后加载，避免把“可能会用到”的代码放进初始包。
- 使用交互边界拆分 chunk，而不是按文件大小机械拆分。

### 2.2 依赖原则
- 重依赖组件（如 `Table/Form/DatePicker/Chart`）优先下沉到懒加载子组件。
- 父组件只保留触发状态与核心数据，不保留低频流程的全部状态。
- 谨慎新增第三方库，优先复用现有组件或轻量实现。

### 2.3 渲染原则
- 减少不必要渲染：稳定 props、提取子组件、必要时用 `useMemo/useCallback`。
- 大列表分页或虚拟化，避免一次渲染大量节点。
- 非关键 UI 可延迟挂载，避免阻塞主线程。

### 2.4 数据原则
- 按需请求，只有功能打开时才发起对应接口。
- 尽量不用缓存。
- 先渲染稳定数据，再异步刷新。
- 如果使用缓存，缓存必须可降级，读取失败不能影响业务主流程。

### 2.5 资源原则
- 图片优先压缩与尺寸匹配，避免过大资源进首屏。
- 清理无用 CSS/JS，避免长期积累的历史包袱。
- 关注总传输体积与关键路径体积，不只看单文件大小。

### 2.6 验收原则
- 用生产模式测性能，不使用 dev 环境分数作为最终依据。
- 每次优化都要保留前后对比（指标、包体、关键交互耗时）。
- 变更要附带复用规则说明，避免只“修一次”。

## 3. 决策清单（做之前先过一遍）
1. 这段功能是否属于首屏必需？
2. 这段功能是否可以在“点击后”再加载？
3. 这段逻辑是否可以迁到子组件，减少父组件状态复杂度？
4. 这次请求是否可延迟、合并或缓存？
5. 是否引入了新的重依赖，能否替代？
6. 优化后是否可通过 `build + test` 且行为不变？

## 4. 标准实施流程（跨页面复用）
1. 建立基线：记录当前 Lighthouse 与包体结果。
2. 定位热点：锁定大包来源、主线程重任务、重复渲染点。
3. 选择策略：优先做懒加载拆分、请求时机调整、渲染减负。
4. 实施改造：小步提交，每一步都可独立回滚。
5. 回归验证：执行 `npm run build`、`npm test`，对比指标变化。
6. 沉淀规则：把可复用做法记录到文档，形成团队规范。

## 5. 常用模式模板

### 5.1 交互触发型懒加载
```jsx
import { Suspense, lazy, useState } from 'react'

const HeavyPanel = lazy(() => import('./HeavyPanel.jsx'))

const [panelOpen, setPanelOpen] = useState(false)

{panelOpen && (
  <Suspense fallback={null}>
    <HeavyPanel onClose={() => setPanelOpen(false)} />
  </Suspense>
)}
```

### 5.2 状态就近收口
- 父组件只持有：`open/type/id` 等触发状态。
- 子组件持有：表单态、筛选态、请求态、提交态。
- 子组件成功后通过 `onClose/onSuccess` 向上通知。

### 5.3 请求按需触发
- 打开弹窗再请求下拉选项与统计数据。
- 切换到低频 tab 再请求 tab 数据。
- 不在首屏预取低频路径数据。

## 6. 指标到动作映射
- `FCP/LCP 高`：先减首屏包体与首屏请求数量。
- `TBT 高`：先拆重组件、减少同步计算、延迟低优先级任务。
- `Unused JS 高`：检查懒加载边界和依赖下沉是否生效。
- `Network payload 高`：压缩资源、移除无用依赖、按需加载。
- `CLS 高`：给媒体和异步区域预留稳定尺寸。

## 7. PR 验收清单
- 是否减少了关键路径代码和请求。
- 是否新增或优化了代码拆分边界。
- 是否避免了无意义渲染与重复请求。
- 是否保持功能行为与文案一致。
- 是否完成 `build + test`，并给出结果。
- 是否补充可复用经验到文档。
- 是否更新组件优化台账（见 `FrontendPerformanceChangeLog.md`）。

## 8. 常见反模式
- 只写了 `lazy`，但重依赖仍在父组件 import。
- 拆了组件，状态和请求却仍集中在父组件。
- 为了省事预取大量低频数据。
- 在 dev 环境做 Lighthouse 后直接下结论。
- 只给“优化后分数”，没有“优化前后对比”。

## 9. 附录：Dashboard 示例（案例，不是规则本体）
- 将批量操作弹窗迁移到独立组件并懒加载。
- 父组件仅保留 `batchModalType`，子组件承接表单与请求。
- 目标是验证“交互边界拆分 + 状态下沉”的通用模式。

## 10. 组件优化记录规范（每次改动必填）
每次做前端性能优化时，必须在 `FrontendPerformanceChangeLog.md` 增加记录，最少包含以下字段：
- 变更日期
- 页面/模块
- 组件或文件路径
- 优化方法（如懒加载、按需请求、延迟渲染、资源精简）
- 目标指标（如 FCP/LCP/TBT/Unused JS）
- 验证结果（如 `npm run build`、`npm test`、Lighthouse 对比）

记录原则：
- 一次 PR 至少新增一条记录。
- 组件级别优先，不只写“某页面优化”。
- 方法描述要可复用，避免只写业务细节。
