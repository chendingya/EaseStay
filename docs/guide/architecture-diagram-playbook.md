# 架构图复用手册（可复用版）

## 1. 目标
- 面向汇报场景的系统架构图，优先保证“远距离可读性”。
- 在不牺牲技术表达的前提下，控制信息密度，避免“字小、留白大、细节过载”。

## 2. 输出标准
- 单页图需在 1366px 宽屏上无缩放可读。
- PPT 插图场景下，模块标题应一眼可辨，正文一句话读完。
- 同一页面最多 3 层信息：标题层、模块层、说明层。

## 3. 信息分层规则
1. 全局层：系统名称 + 一句话说明（不写实现细节）。
2. 分层层：按端或按层划分（如“移动端/管理端/服务端”）。
3. 模块层：每个模块只保留
   - 模块名
   - 1 句职责
   - 2-3 个关键词标签

## 4. 文案压缩规则
- 每个模块说明控制在 14-24 个汉字。
- 禁止在图中塞入路径、接口列表、字段细节（可在备注文档补充）。
- 优先“职责语义”而非“代码细节”：
  - 推荐：`订单生命周期与库存可用性控制`
  - 不推荐：`services/roomAvailabilityService.js + pricingService.js`

## 5. 可读性排版基线
- 页面标题：`2.4rem - 3.8rem`
- 分层标题：`1.1rem - 1.3rem`
- 模块标题：`1.5rem - 1.8rem`
- 模块正文：`1.15rem - 1.3rem`
- 标签文字：`1rem` 左右
- 行高：正文 `1.45 - 1.58`

## 6. 留白与尺寸规则
- 禁止硬性大高度卡片：`min-height` 默认设为 `0` 或不设置。
- 用 `padding` 控制呼吸感，不靠“空白高度”凑视觉。
- 栅格间距建议 `10px - 14px`，避免过密或过散。

## 7. 视觉风格规则（浅色多巴胺）
- 背景：浅色渐变 + 低透明光斑，避免纯白刺眼。
- 卡片：白底轻阴影，使用高饱和描边作为功能分组提示。
- 强调色建议：
  - 青蓝（接入/数据流）
  - 橙色（业务/运营）
  - 绿色（安全/稳定）
  - 洋红（核心链路/关键能力）

## 8. 响应式规则
- 桌面端：三列或双列布局。
- 移动端：统一单列，字号不降级（必要时升一级）。
- 断点下优先保证可读，不强求“同屏展示全部模块”。

## 9. 交付检查清单
- 是否存在“大片空白 + 小字体”？
- 模块说明是否都可在 2 秒内读完？
- 是否出现过多实现细节（路径、脚本、参数）？
- 模块颜色是否有语义，不是随机上色？
- 缩放到 PPT 常用比例后是否仍清晰？

## 10. 本项目复用建议
- 三端总览图：用于开场与全局关系说明。
- 分体架构图：用于逐端深讲（移动端/管理端/服务端）。
- 技术细节页：单独补充，不塞进总览图。
