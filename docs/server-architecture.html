服务端架构图

左侧看工程分层与代码目录，右侧看请求从入口到数据层的真实执行路径。

代码结构视图

1) 启动与应用壳

启动文件加载环境变量并拉起 Express 应用。

server/src/index.js server/src/app.js
↓

2) 路由聚合层

API 入口统一聚合业务路由，按领域拆分 route 文件。

server/src/routes/index.js server/src/routes/auth.js server/src/routes/user.js server/src/routes/adminHotels.js
↓

3) 鉴权与权限中间件

Token 校验与角色权限控制，保护商户/管理员专属接口。

server/src/middleware/auth.js authRequired requireRole
↓

4) 控制器与服务层

Controller 处理入参和响应，Service 承载库存、价格、通知等业务逻辑。

server/src/controllers/*.js server/src/services/roomAvailabilityService.js server/src/services/pricingService.js
↓

5) 数据与外部依赖层

Supabase 为主数据存储，地图与图片代理能力通过服务封装接入。

server/src/config/supabase.js server/src/services/mapService.js server/src/routes/image.js server/schema.sql
质量保障：`server/__tests__` 覆盖路由、控制器、中间件与核心服务， 架构目标是“路由薄、服务厚、权限清晰、依赖可替换”。

运行时数据流视图

A. 标准请求主链路
HTTP 请求进入→ Express 路由匹配→ authRequired / requireRole→ Controller→ Service→ 数据层响应
B. 登录鉴权链路
POST /api/auth/login→ authController→ authService 校验→ JWT 生成→ 返回 token + role
C. 下单库存链路
POST /api/hotels/:id/orders→ publicHotelsController→ roomAvailabilityService→ pricingService→ 订单写库
D. 地图与图片链路
/api/map/* 或 /api/image→ mapController / imageController→ 外部服务调用→ 结果清洗与白名单控制→ 统一响应
部署落地：`docker-compose.yml` 中 `server(4100)` 对外提供 API， 与 `admin(4101)`、`mobile(10086)` 形成三端协作闭环。
